Practical setup

Run it locally first. Push the same service to Fly when you are ready.

These notes mirror the current repo state: the server is a Rust binary, the frontend is served statically by the server, and Fly deployment uses the included Dockerfile and `fly.toml`.

Local start Fly deploy GitHub Actions Local script Env vars
01

Local start

Start the server from the repo root and open the app in the browser.

cargo run -p mermaduckle-server
  • Marketing site: /
  • Control plane: /app
  • Deploy docs: /docs
  • Health endpoint: /api/health
02

Deploy to Fly.io

The repo already includes the deployment artifacts this command expects.

flyctl deploy -a <app-name> --dockerfile deploy/Dockerfile.fly

Current config in this repo uses app = "mermaduckle" and exposes the Rust server on internal port 3001.

03

GitHub Actions deploys

Use an app-scoped deploy token and let pushes to main ship to Fly.

fly tokens create deploy --app <app-name>
  • Add the returned token as the GitHub Actions secret FLY_API_TOKEN.
  • The repo workflow file is .github/workflows/fly-deploy.yml.
  • The workflow runs flyctl deploy --remote-only --config fly.toml.
04

Local helper script

Use the included PowerShell script when you want to deploy from your machine.

pwsh -File tools/deploy_fly.ps1 -AppName mermaduckle -RemoteOnly

Add -BuildOnly if you only want to push the image without releasing it yet.

05

Domain and certificate workflow

These are the commands described in the deployment guide.

flyctl domains add mermaduckle.com -a <app-name>
flyctl ips allocate-v4 -a <app-name>
flyctl certs create mermaduckle.com -a <app-name>
