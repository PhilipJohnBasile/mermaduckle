Self-hosted guide

Self-host the same product you see in the public sandbox.

This domain hosts three things: the marketing site, a public sandbox at /app, and this deployment guide. Customer production environments use the same Rust service, but run as separate deployments under your ownership or as a supported rollout.

02

Local start

Run a local instance from the repository root and inspect every public route.

cargo run -p mermaduckle-server
  • / marketing site
  • /app sandbox UI
  • /docs deployment guide
  • /api/health service readiness endpoint
03

Deploy to Fly.io

The repository already includes the Dockerfile and fly.toml required for a Fly rollout. 

flyctl deploy -a <app-name> --remote-only --config fly.toml

The current repo is configured around a single service that serves the marketing site, sandbox, docs, and API from internal port 3001.

04

GitHub Actions deploys

Push to main to trigger Fly deployment after CI passes.

fly tokens create deploy --app <app-name>
  • Add the token as the repository secret FLY_API_TOKEN.
  • The deployment workflow lives in .github/workflows/fly-deploy.yml.
  • The workflow runs flyctl deploy --remote-only --config fly.toml.
05

Domains and TLS

Use these Fly commands when mapping a production domain and issuing certificates.

flyctl domains add mermaduckle.com -a <app-name>
flyctl ips allocate-v4 -a <app-name>
flyctl certs create mermaduckle.com -a <app-name>
