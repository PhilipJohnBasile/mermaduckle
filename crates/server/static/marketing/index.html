Why Mermaduckle

Built for operations, not just prototyping.

Many platforms are optimized either for graph design or for connector volume. Mermaduckle is designed for the moment an AI workflow touches production systems and needs approval, traceability, runtime ownership, and a clean deployment path.

Workflow builders

Strong for rapid assembly

Useful when the priority is composing prompts, chains, and agent graphs quickly.

  • Fast visual design for AI flows
  • Great for experimentation and prototyping
  • Operational controls are often a secondary concern
Automation platforms

Strong for integration breadth

Useful when the priority is moving events and data across many external systems.

  • Broad connector coverage
  • Good for SaaS-heavy automation workloads
  • AI governance is usually not the product center
Mermaduckle

Designed for production control

A governed control plane for AI workflows that need review, traceability, runtime settings, and an operating model that can survive real teams and production traffic.

  • Approval checkpoints and audit events in the core product
  • Workflow, agent, and run visibility in one place
  • Flexible deployment across self-hosted and managed environments

Governance

Human review, traceability, and recovery are built into the product.

Mermaduckle includes the operating controls teams need once workflows affect external systems: a pending approvals queue, audit events, reporting endpoints, and recovery paths tied to the run history itself.

01

Approval checkpoints

Pause at pending_approval and require an explicit decision before continuing.

02

Audit trail

Track workflow, approval, and agent activity with event types, timing, and outcome metadata.

03

Recovery workflow

Use logs and context to patch and retry runs without losing the operating record.

Deployment guide

Ship the same service from local development to production.

The repository includes a Fly Dockerfile, a working fly.toml, a GitHub Actions deployment workflow, and a PowerShell helper script so teams can move from evaluation into customer-owned deployment without changing stacks.

Deploy to Fly Open the deployment guide 
flyctl deploy -a mermaduckle --remote-only --config fly.toml

Push to main for GitHub-driven deployments, or use the included helper script for manual releases.

Included in the repository

Everything required for a production rollout

  • deploy/Dockerfile.fly for multi-stage Rust builds
  • fly.toml with HTTP and TLS ports plus health checks
  • .github/workflows/fly-deploy.yml for push-to-main deployment
  • tools/deploy_fly.ps1 for local production releases
  • /api/health for deployment verification
